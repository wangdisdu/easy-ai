# easy-ai 企业级 Harness Agent 系统设计

> 本文是 easy-ai 作为企业级 harness agent 的**系统级**设计，面向架构评审与团队对齐。
> 偏"系统是什么、为什么这样、各部件如何协作"，不包含表结构、SQL、代码伪码等实现细节。
> 实现细节请分别参阅：
> - `long-session-design.md`（长会话 + Todo 可视化 + 反馈闭环）
> - `tool-approval-and-acl-design.md`（工具审批与 ACL）
> - `agent-memory-design.md`（记忆系统）
> - `skill-evolution-design.md`（技能演化）

## 1. 背景与目标

### 1.1 从"能用"到"企业能用"

easy-ai 现阶段具备的 agent 能力来自 deepagents 的基础栈：模型调用、工具编排、技能（SKILL.md）注入、子 agent。这一栈足够做 demo 和内部试点，但要真正进入企业业务生产，离"可运营、可审计、可成长"还差一层：

| 企业诉求 | 缺失的能力 |
|---|---|
| "昨天的对话思路能接着跑吗？" | 跨会话工作记忆 / 长上下文 / 崩溃恢复 |
| "agent 现在在干啥，还剩几步？" | 任务清单实时展示、进度透明 |
| "这个回答不好我想投诉" | 反馈闭环与质量数据沉淀 |
| "agent 用这个工具要走审批" | 工具治理、HITL、风险分级 |
| "agent 记得我是谁吗？" | 跨会话长期记忆 |
| "某个 skill 上线后越用越差" | 使用遥测、回归评估、自动演化 |
| "上线要留痕、出问题要回滚" | 生命周期管理、审计、灰度、合规 |

### 1.2 目标

构建一套 **harness agent 中台能力层**，在 deepagents 之上补齐四大企业能力域（Session / Policy / Memory / Skill），并提供统一的运行上下文、观测、治理、分期交付路径。最终的 agent 应用能像业务系统一样：

- **可追踪**：每一次 agent 运行、工具调用、记忆访问都可追溯到人、会话、策略、版本
- **可控制**：敏感工具走审批、危险操作可拦截、紧急情况可破窗
- **可成长**：skill / memory 随使用演化，越用越准
- **可恢复**：中途中断可续跑、审批后可恢复执行
- **可回滚**：任何策略 / 技能变更都有版本与证据链

## 3. 核心概念模型

### 3.1 实体与关系

```
User ──∈──▶ Team ──∈──▶ Org (Tenant)
 │
 ├─ 创建 ─▶ Conversation ──绑定──▶ App
 │              │                  ├──绑定──▶ Skill (variants / overrides)
 │              ▼                  └──绑定──▶ Tool (policy)
 │           Message
 │
 ├─ 个人记忆 ──▶ Memory (user scope)
 │
 └─ 审批身份 ──▶ ApprovalChain / Delegation
```

### 3.2 关键运行时抽象

| 抽象 | 含义 | 生命周期 |
|---|---|---|
| **AgentRunContext** | 一次 agent 执行的统一上下文（run_id / app_id / user_id / conversation_id / thread_id / tenant_id / team_ids / role_codes / trace_id） | 一次请求 |
| **Conversation** | 业务会话 | 跨多次 run |
| **Thread** | LangGraph checkpoint 的恢复单元，与 Conversation 1:1 | 跟随会话 |
| **Run** | 一次完整的 agent 执行（用户一次提问 → agent 结束或中断） | 单次 |
| **ToolCall** | 一次工具调用；幂等键 `(run_id, tool_call_id, policy_version)` | 单次（可 resume） |
| **Skill Activation** | 本次 run 中某 skill 是否真正被 agent 用上 | 单次 run |

### 3.3 AgentRunContext —— 全链路唯一上下文

所有四大能力域共享同一个 `AgentRunContext`，由入口层（ConversationService / OpenAPI）统一构造，贯穿 Session / Policy / Memory / Skill / Observe。

**关键约束**：
- `AgentRunRequest`（前端传参）**不携带** `thread_id / run_id / tenant_id` 等运行态字段
- 所有中台服务从 `AgentRunContext` 取字段，禁止从历史 `RequestContext` 生造
- `run_id` 贯穿审批幂等、技能遥测、审计记录、观测追踪，是全链路对账的主键

                                          │

**关键依赖链**：
- **HITL 审批 → Session**：审批暂停必须建立在可恢复的运行态之上，没有长会话能力就没有可行的人工干预
- **Skill 演化 → Skill 遥测 → AgentRunContext**：没有稳定的运行标识就没有 A/B 归因和失败分析
- **Memory 语义召回 → 模型网关**：外部依赖，必须有降级路径
- **反馈闭环 → 观测追踪**：差评必须可追踪到具体一次运行才有价值

│  L5 表现层                                                │

├──────────────────────────────────────────────────────────┤

├──────────────────────────────────────────────────────────┤
│  L3 能力服务层                                            │
│  Session · Policy · Approval · Memory · Skill ·           │
│  Feedback · Audit · Notification                          │
├──────────────────────────────────────────────────────────┤
│  L2 Agent 运行时层                                        │
│  记忆注入 · 工具治理 · Todo 广播 · 反思抽取 · 技能遥测     │
│  + 工具执行兜底层（保证任何调用路径都被治理覆盖）           │
├──────────────────────────────────────────────────────────┤
│  L1 数据层                                                │
│  业务库 · 向量库 · 运行态快照 · 对象存储                  │
└──────────────────────────────────────────────────────────┘
```

**分层原则**：

- **能力服务层**是业务逻辑的唯一出入口；运行时层只做挂接，不持久化
- **工具执行兜底**：任意工具调用路径（主 agent、子 agent、task 转调等）最终都会经过同一层治理检查，不给绕过留后门
- **数据层按职责拆分**：业务数据、运行态、向量、审计各有其存储，互不重建

## 6. Session Layer —— 长会话、Todo、反馈

### 6.1 设计定位

Session Layer 解决的是一个根本性的信任问题：用户与 agent 的交互不该像"无记忆的问答"，而应该像"有持续性、可观察、可评价的协作"。
这一层把一次次离散的对话粘合成**可恢复、可追问、可反馈**的会话资产，是所有上层能力（审批、记忆召回、技能遥测）能站得住脚的时间轴基础。

**它对用户的承诺**：
- 昨天的对话今天接着讲，agent 的思路不会丢
- 任务执行到哪一步看得见，不用盯屏等
- 答案不满意可以打分、可以留言，团队能看到

**它对系统的承诺**：
- 提供稳定的会话 / 运行标识，作为审批幂等、技能归因、审计串联的锚点
- 把运行态与业务态严格分层，避免上层功能误用对方的数据
- 提供可恢复的暂停点，让人工审批、破窗、事故排查成为可能

### 6.2 底座：Checkpoint 机制

整个 Session Layer 最核心、最底层的东西是 **Checkpoint（运行态快照）**。把 Checkpoint 拿掉，这一层的所有能力——以及上层的审批、审批恢复、崩溃续跑——都会立刻塌掉。先讲清楚 Checkpoint 是什么，再讲其他子系统才有意义。

#### 6.2.1 Checkpoint 存了什么

Agent 在跑的时候，脑子里有一摊东西是"工作现场"：

| 内容 | 举例 |
|---|---|
| 消息列表 | 用户说过什么、agent 回答过什么、工具调用过什么 |
| 任务清单（todos） | "查数据 → 做表 → 发邮件"，每一项的完成状态 |
| 虚拟工作文件 | agent 过程中产出或参考的临时文件（报告草稿、数据片段、技能文本） |
| 工具调用中间态 | 某个工具调到一半还没返回，参数和调用位置都要记住 |
| 模型思考轨迹 | 推理链、scratch 区域 |

这些合起来叫 **agent 的运行态**。Checkpoint 就是**把这摊东西在每一步之后完整存盘的机制**。

#### 6.2.2 Checkpoint 怎么组织

Checkpoint 有三个关键属性：

- **线程编号（thread id）**：每个会话有一个唯一的线程编号（通常就是会话 id 的字符串形式），同一会话的所有快照共享这个编号。它是"找到上次存档"的钥匙。
- **版本与父子链**：同一线程下的快照按时间形成父子链，每份快照知道自己的上一份是谁，方便回溯、分叉、审计。
- **增量写入**：每份新快照只记录变化，不复制整份历史，避免长会话存储爆炸。

```
  线程编号："1001"

   T1 用户发第 1 句 ────▶ 快照 v1 ◀──┐
                                     │ 父子链
   T2 agent 回复    ────▶ 快照 v2 ◀──┘
                                     │
   T3 工具调用开始  ────▶ 快照 v3 ──┐
   T4 工具返回      ────▶ 快照 v4 ◀┘

   ...每一步 agent 推进都写一份新快照...
```

#### 6.2.3 什么时候写、什么时候读

**写入时机**：每当 agent 完成一步有意义的动作——收到模型回复、调完一个工具、更新完待办——运行态都会**自动增量存盘**。这是"过程级"的持久化，不是"整次运行结束才写一次"。

**读取时机**：新一轮对话到来时，系统用这次对话所在的线程编号，去快照存储里找**最新那份**快照，把上次的运行态整个恢复到内存，再把本轮的新输入接着往下跑。

```
新消息到来
    │
    ▼
 用线程编号 "1001" 查快照存储
    │
    ├─▶ 找到最新快照 → 还原消息、todos、文件、工具中间态
    │
    └─▶ 找不到（首次对话 / 被清理 / 损坏）→ 走降级路径
```

#### 6.2.4 会话与 Checkpoint 的绑定

业务会话和线程编号绑死：

```
业务会话表
├── 会话 id：1001
├── 用户：  张三
├── 绑定应用：客服 Agent
└── 线程编号："1001"    ← 会话创建时落地，之后永不变
```

**一个会话对应一条线程上的一连串快照**，关系稳定、可审计。

#### 6.2.5 消息"只传新的一句"的输入协议

因为 Checkpoint 恢复时已经把整段历史还原好了，如果业务层再把历史全传一遍，会出现"历史翻倍"（token 费用翻倍、上下文浪费、压缩和工具配对错乱）。所以有条必须遵守的规则：

| 是否启用长会话 | 业务层传给 agent 的消息 |
|---|---|
| 未启用 | 整段历史（老行为） |
| 已启用、正常路径 | **只传本轮新的这一句**，历史由 Checkpoint 自动恢复 |
| 已启用、快照缺失 | 走降级重建（见下一条） |

#### 6.2.6 Checkpoint 缺失时的降级重建

Checkpoint 可能丢失或损坏——用户主动清空过、存储被清理过、迁移出过问题。不能让用户的历史会话彻底失忆，所以准备一条降级路径：

```
 查不到 Checkpoint
    │
    ▼
 从业务消息表捞最近 N 条消息
    │
    ▼
 用这 N 条消息拼出一个"最小可运行状态"写入新快照
    │
    ▼
 再把本轮新消息作为正常输入继续跑
    │
    ▼
 前端提示"历史已降级恢复，之前的工作文件和待办不可用"
 留一条 checkpoint_rebuilt_from_messages 审计事件
```

降级状态只恢复消息文本，**不恢复** todos、虚拟文件、工具中间态——这些东西业务消息表本来就没有。

#### 6.2.7 为什么它是整层的地基

去掉 Checkpoint，Session Layer 的各个能力会连锁塌陷：

| 能力 | 如果没有 Checkpoint |
|---|---|
| 跨会话恢复 | 直接不可能 |
| "只传新消息"输入协议 | 失去存在意义（没地方恢复历史） |
| 任务进度实时同步 | 当轮还能看，切回历史会话时 todos 全丢 |
| 人工审批（Policy Layer） | 没法实现，因为审批需要 agent 在执行中暂停再继续 |
| 审批恢复续跑（Policy Layer） | 没法实现，恢复就是"从快照接着跑" |
| 崩溃续跑 | 进程挂了就整轮对话作废 |
| 反馈定位 | 差评仍能打分，但难追溯到当时 agent 的中间决策过程 |

这也解释了为什么路线图里 Session 必须先于 Policy 审批层交付——Checkpoint 是所有上层信任能力的地基。

- **长会话引擎**：围绕 Checkpoint 的工程化包装——管理快照读写、把业务会话和线程编号绑定、提供降级重建路径、在运行态过长时触发上下文压缩。Checkpoint（§6.2）是这个子系统的底座，其他几件事都是让 Checkpoint 能在企业场景稳定工作的辅助能力。
- **任务进度实时同步**：Agent 处理任务时会自己列一份"待办清单"（比如"查数据 → 做表 → 发邮件"），这个子系统把清单每一步的变化实时推到用户屏幕。清单本身是 agent 运行态的一部分，存在 Checkpoint 里；切回历史会话时也能还原上次的状态。
- **反馈中心**：每条 agent 消息可打分、可打标签、可留评论；反馈绑定到这次运行的观测追踪，形成"差评 → 原始 trace → 涉及的 skill / 工具 / 记忆"的完整链路。
- **会话生命周期**：会话从创建到归档的完整状态机，控制运行态存活期、业务消息保留期、反馈累积窗口。

### 6.4 能力清单

| 能力 | 目的 | 对 Checkpoint 的依赖 |
|---|---|---|
| **跨会话恢复** | Agent 运行态跨多次对话保留 | 直接依赖：恢复就是"读 Checkpoint" |
| **上下文压缩** | 超长会话自动摘要，控制 token 成本与首 token 延迟 | 直接依赖：压缩后结果写回 Checkpoint |
| **消息输入协议** | 启用长会话时只传本轮新消息，避免历史重复注入 | 直接依赖：输入协议为"让 Checkpoint 正确恢复历史"而存在 |
| **任务进度实时同步** | 任务清单逐步推给前端，用户随时看到做到哪一步 | 间接依赖：todos 存在 Checkpoint 里，历史会话还原靠它 |
| **反馈闭环** | 每条回复可 👍/👎 + 标签 + 评论 | 弱依赖：有 Checkpoint 时差评可追溯到中间态 |
| **会话重置 / 恢复** | 用户一键清空运行态；损坏时从业务消息表重建 | 直接依赖：操作对象就是 Checkpoint |

### 6.5 核心数据分层

```
┌────────────────────────────────────────────────────┐

│  业务消息表：用户可见的消息列表、追踪 id、反馈绑定 │

├────────────────────────────────────────────────────┤
│  运行态事实源                                      │
│  Checkpoint 存储：消息、todos、虚拟工作文件、      │
│  工具调用配对、模型思考等 agent 决策链组成          │
│  权威性：运行恢复的唯一依据；不对外、可清理          │
├────────────────────────────────────────────────────┤
│  反馈与追踪                                        │
│  反馈表：与业务消息 1:N（同一消息多轮反馈）         │
│  追踪 id：绑定到业务消息，可反向定位到运行态和涉及  │

```

**职责切分原则**：两套事实源独立、互不重建。仅在 Checkpoint 缺失或损坏时，允许从业务消息重建最小可运行状态（只恢复消息，不恢复 todos / 虚拟文件 / 工具中间态），并留审计事件、前端明示"历史已降级恢复"。

```

    │

### 6.7 生命周期

**会话状态机**：

```
  新建 ──▶ 活跃 ──▶ 长期静默 ──▶ 用户归档 ──▶ 冷归档 ──▶ 回收
                              │
                              └──▶ 被遗忘权 ──▶ 物理抹除
```

**Checkpoint 生命周期**（独立于会话生命周期清理）：

- **写入**：agent 每推进一步，自动增量写入
- **保留**：跟线程编号绑定；会话归档后快照保留一段观察期（支持回看、追诊）
- **清理**：过期后自动清理，**只清快照，不影响业务消息与反馈**
- **被遗忘权触发**：按用户抹除时，线程下所有快照一并物理删除

| Policy Layer | 审批中断 / 恢复直接建立在 Checkpoint 之上：中断就是"暂停后保存 Checkpoint"，恢复就是"按线程编号读 Checkpoint 继续跑" |
| Memory Layer | 情景记忆（每会话一条摘要）在会话归档时由本层触发生成 |
| Skill Layer | 技能遥测按运行标识关联本层；反馈数据成为技能差评回流的源头 |

## 7. Policy Layer —— 工具审批与 ACL

### 7.1 设计定位

Policy Layer 是 agent 对外产生真实世界影响时（发邮件、改数据库、调外部 API）的**唯一治理入口**。企业不能接受"模型说调什么就调什么"，必须有一套结构化机制回答三个问题：

- **能不能调**（准入）：当下这位用户、这个参数、这个时间点，是否允许调这个工具
- **要不要人审**（审批）：什么风险档次的调用需要暂停等人工确认
- **出了事查得清吗**（审计）：谁决定了放行、谁批了、输入输出是什么、有没有被篡改

这一层的核心定位不是"加个开关允许或拒绝"，而是把工具调用纳入**企业风控的控制论闭环**：策略规则 → 运行时准入 → 人工审批 → 审计追溯 → 策略调优。

### 7.2 内部子系统

```
┌──────────────────────────────────────────────────────────┐
│                     Policy Layer                          │
│                                                           │
│  ┌────────────────┐  ┌────────────────┐  ┌────────────┐  │

│  │ ·多维度条件    │  │ ·工具默认风险 │  │ ·工具 schema│  │
│  │ ·优先级评估    │  │ ·参数启发式   │  │  哈希校验   │  │
│  │ ·影子 / 演练   │  │ ·策略强制覆盖 │  │ ·漂移检测   │  │
│  │ ·模板化        │  │               │  │ ·自动挂起   │  │
│  │ ·版本化        │  │               │  │            │  │

- **策略引擎**：给定"工具 + 参数 + 用户身份 + 环境"，回答允许 / 拒绝 / 需审批；支持影子演练模式（只记录不阻断）
- **风险评估**：对无策略命中的工具调用按默认风险 + 参数启发式打分，驱动"应走哪一级审批"
- **指纹守护**：防止工具定义被悄悄修改（MCP server 替换、API schema 偷改），配套工具评审流程
- **审批编排**：管理审批单全生命周期，涵盖多级审批链、SLA 升级、休假委托、预审批与破窗
- **内容安全**：输入脱敏用于审计（不改实际执行参数）、输出 DLP 扫描、防止工具输出里的指令诱导 agent
- **通知派发**：审批到来、升级、破窗、指纹不匹配等事件分发给对应渠道，支持外部工作流回调
- **审计流**：所有决策事件以追加写入的形式入审计库，带签名链，合规可导出、可离线验证

### 7.3 能力清单

| 能力 | 目的 |
|---|---|
| **策略准入** | 多维度条件评估（用户 / 组 / 工具 / 参数 / URL / 时间 / 租户） |
| **风险分级** | LOW / MED / HIGH 自动打分，决定放行 / 审批 / 多级审批 |
| **人工审批** | 高风险工具调用暂停 → 审批 → 恢复执行 |
| **预审批** | 一次通过覆盖未来 N 次 / 整个会话 / 当天 同类调用 |
| **审批链与升级** | 按 SLA 自动升级到上级；休假自动委托 |
| **破窗应急** | 超级管理员在事故态可强制放行，附强审计 |
| **工具指纹** | 防工具定义替换 / 偷改导致策略失配 |
| **内容安全** | 输入脱敏 / 输出扫描 / 指令注入检测 |
| **策略治理** | 版本化 / 影子模式 / 模板 / 变更冷却 + 双人核准 |
| **通知与外部集成** | 站内 / Slack / 邮件 / Webhook / 工单系统 |

### 7.4 核心流程

```

    │
    ├─ 多维度策略评估（支持影子 / 演练模式）
    │
    ├─ 拒绝：移除该调用，回传拒绝说明给模型
    │
    ├─ 需审批：按幂等键复用或创建审批单
    │
    └─ 收齐需审批的调用 → 暂停运行
        │
        ▼
    运行态快照保存，通知前端进入等待态，原流结束
        │
        ▼
    审批人：通过 / 拒绝 / 紧急放行 / 参数改写
        │

        │

        │

        ├─ 真实调用
        └─ 输出安全扫描 + 审计留痕
```

### 7.5 风险分级模型

一次工具调用的风险等级由三个来源取最大值决定：

```
风险等级 = max(
    工具默认风险（工具定义时标注：读类 LOW，写类 MED，不可逆 HIGH）,
    策略命中时的显式覆盖（策略可把风险抬高或降低）,
    参数启发式（金额阈值、SQL 关键字、外发域名等触发 HIGH）
)
```

风险等级映射到处置动作：

| 风险 | 处置 |
|---|---|
| LOW | 若无策略拒绝则直接放行 |
| MED | 需单级审批（通常是直属主管或 app 管理员） |
| HIGH | 需审批链多级通过；关键操作还叠加破窗机制 |

### 7.6 策略评估模型

策略是一组有优先级的规则，每条规则包含若干条件和一个处置动作：

- **条件维度**：身份（用户 / 组 / 角色）、资源（工具 id / 源 / 全限定名）、参数（匹配、正则、数值边界）、网络（URL / 域名白黑名单）、上下文（时间窗口 / 工具链前序 / 租户）
- **处置动作**：允许 / 拒绝 / 走审批 / 演练记录
- **评估次序**：按优先级从高到低匹配，第一条命中决定结果；全部未命中时取默认策略
- **模式**：
  - **正式（active）**：命中即生效
  - **影子（shadow）**：命中只记录，不改变决策，用于新策略上线前观察命中面
  - **演练（dry-run）**：全局开关；所有策略只记录不阻断，用于策略库整体升级时

策略支持**模板化**（通用场景一键复制）、**版本化**（每次变更保留快照可回滚）、**变更冷却**（关键策略改动要双人核准 + 生效延迟）。

### 7.7 审批人体系

**审批链**是对一个风险等级可配置的多级审批路径，每级配置：

- 审批人类型：具体用户 / 角色 / 用户组 / 发起人的直属上级
- SLA 超时：未响应自动升级到下一级
- 最终级失败处置：最终超时视为拒绝 / 或视为通过（仅低风险可配）

**委托**解决审批人休假 / OOO 问题，可按时间段或特定 app 转交。**防自审**由系统强制，发起人绝不会成为自己调用的审批人。

**预审批**是对"agent 将连续执行多次同类调用"场景的优化：审批人首次通过时可勾选"本会话 / 本日 / 未来 N 次同类参数模式"，后续相同模式的调用直接放行，只留审计。

**破窗**是紧急事故的应急通道：独立权限角色（不与普通管理员合并），要求详细的事故说明 + 双人确认，事件以最高严重度推送到所有通知渠道，周度自动汇总给合规复盘。

### 7.8 审批单状态机

```
  接收到需审批请求
          │
          ▼
       pending ────[SLA 超时]────▶ escalated ──▶ pending（下一级）
          │
          ├──[审批人通过]────▶ approved ──▶ 由恢复服务续跑
          ├──[审批人拒绝]────▶ rejected ──▶ 回传拒绝给 agent
          ├──[发起人取消]────▶ cancelled
          ├──[破窗放行]─────▶ break_glass ──▶ 等同 approved，但审计标注
          └──[所有级都超时]──▶ timeout ──▶ 回传拒绝给 agent
```

所有状态迁移都留审计事件；幂等键保证重复触发不会创建重复审批单。

### 7.9 数据分类与所有权

| 数据类别 | 内容 | 所有权 | 保留 |
|---|---|---|---|
| 策略定义 | 规则、版本、模板 | 组织 / 应用管理员 | 版本永久保留 |
| 审批单 | 状态、审批人、意见、上下文快照 | 运行时自动生成 | 合规周期不删 |
| 预审批令牌 | 范围、有效期、剩余次数 | 审批人授予 | 过期自动失效 |
| 委托记录 | 委托人 / 受托人 / 时段 / 范围 | 委托人自主 | 过期归档 |
| 工具指纹 | 哈希、评审状态、信任时间 | 组织管理员 | 随工具生命周期 |
| 审计流 | 策略决策、审批、破窗、指纹事件等 | 合规只读 | 按合规要求长保 |

### 7.10 与其他层的协作

| 其他层 | 协作关系 |
|---|---|
| Session Layer | 人工审批的暂停-恢复直接建立在长会话之上；审批恢复服务共用会话锁 |

| Skill Layer | 发布高权限技能（含高风险工具）需要强制过策略评审；技能本身的变更也走类审批流程 |

## 8. Memory Layer —— 企业级长期记忆

### 8.1 设计定位

Memory Layer 解决的是 agent 的"长期知情"问题：用户不应该每次都重复自我介绍，agent 不应该忘记上次讨论的结论，团队不应该每个人都从零给 agent 喂一遍业务术语。

**它与其他"知识 / 上下文"类能力的分工**：

| 能力 | 时间尺度 | 内容性质 | 是谁的知识 |
|---|---|---|---|
| 运行态快照（会话层） | 单次会话内 | 当前执行中的工作记忆 | 这次运行自己的 |
| **Memory（本层）** | **跨会话长期** | **关于用户 / 团队 / 应用的事实与偏好** | **你和团队的** |
| Skill（技能层） | 长期稳定 | 做某类任务的方法 | 应用的 |
| RAGFlow（企业知识库） | 长期稳定 | 企业文档 / 手册 | 组织的 |

这一层既要**够用**（真的能召回过去说过的事、记住偏好），又要**可控**（用户能看、能改、能删、不被滥用、符合合规）。

### 8.2 内部子系统

```
┌────────────────────────────────────────────────────────┐
│                    Memory Layer                         │
│                                                         │
│  ┌─────────────────┐  ┌─────────────────┐              │
│  │ 注入引擎         │  │ 反思抽取         │              │
│  │                 │  │                 │              │
│  │ · 多层合并       │  │ · 对话后自动抽   │              │
│  │ · 按预算裁剪     │  │   候选事实       │              │
│  │ · 降级路径       │  │ · 冷启动阈值     │              │
│  │ · 证据回链       │  │ · 成本配额       │              │
│  └─────────────────┘  └─────────────────┘              │
│                                                         │
│  ┌─────────────────┐  ┌─────────────────┐              │
│  │ 写入工具集       │  │ 后台维护         │              │
│  │                 │  │                 │              │
│  │ · remember      │  │ · 情景摘要       │              │
│  │ · forget        │  │ · 去重合并       │              │
│  │ · recall        │  │ · 置信度衰减     │              │
│  │ · list          │  │ · 过期清理       │              │
│  │ （受策略守护）   │  │                 │              │

│                                                         │
│  ┌─────────────────┐  ┌─────────────────┐              │
│  │ 用户确认界面     │  │ 治理与合规       │              │
│  │                 │  │                 │              │
│  │ · 待确认候选     │  │ · PII 分级       │              │
│  │ · 已记忆清单     │  │ · 被遗忘权       │              │
│  │ · 证据链跳转     │  │ · 数据携带权     │              │

```

- **注入引擎**：在每次对话前，按作用域层级和 token 预算，把当前最相关的记忆片段合并进 system prompt
- **反思抽取**：每轮对话后由轻量模型扫最近消息，抽取可能值得记住的事实作为候选
- **写入工具集**：暴露给 agent 的标准记忆操作；所有调用都受策略层治理（能不能写 / 写什么 / 是否需要用户确认）
- **后台维护**：定期执行情景摘要、相似记忆合并、低置信度记忆衰减、过期清理
- **用户确认界面**：把"待确认候选"、"已记忆"、"被引用次数"透明化给用户，用户是记忆的最终仲裁者
- **治理与合规**：PII 分级、被遗忘权、数据导出、租户隔离等企业合规要求

### 8.3 五层记忆分类

五层不是"同一个东西的五种实现"，而是**五种不同形态的知识**，共同组成 agent 对"过去"的全部认知。每一层有独立的数据形态、写入方式、读取方式和典型用途。

#### 8.3.1 总览

| 层 | 名称 | 形态 | 写入者 | 召回方式 | 典型条目 |
|---|---|---|---|---|---|
| L1 | 静态文件记忆 | Markdown 文件，按组织 / 团队 / 应用 / 用户分层 | 管理员 / 用户 | 固定全量注入 | "本项目 API 命名用单数" |
| L2 | 结构化 KV | 键值对 + 类别 + 置信度 | 用户 / 管理员 / agent（须确认） | 按作用域合并注入 | `language_preference = 中文` |
| L3 | 语义记忆 | 自由文本 + 向量 | 用户 / 管理员 / agent（须确认） | 向量相似度检索 | "用户说过不喜欢表格，偏好 CSV" |
| L4 | 情景记忆 | 每会话一条自动摘要 + 向量 | 系统后台 | 向量检索历史会话 | "4-20 讨论促销活动 X，倾向方案 B" |
| L5 | 反思记忆 | Agent 自动抽取的候选事实 | Agent（默认待确认） | 用户确认后晋升到 L2/L3 | "用户偏好代码块回答编程问题" |

#### 8.3.2 L1 —— 静态文件记忆（AGENTS.md）

**定位**：结构化的团队 / 组织 / 个人背景知识，以 markdown 形式存在，按作用域分路径组织。

**典型内容**：术语表、技术栈惯例、流程约定、应用角色定位、用户个人档案。

**关键特性**：
- 内容不会随对话产生，是事先约定好的
- 每次对话前整份加载进 system prompt（因为内容短且固定，不需要检索）
- **Agent 不能写**——管理员写 org / team / app 级，用户写 user 级；Agent 的自动记忆走 L5
- 手动维护，有版本与修改记录

**类比**：像项目的 README，新成员看一遍就基本掌握背景。

#### 8.3.3 L2 —— 结构化 KV 记忆

**定位**：可以标准化的**原子级事实**，按 `键 → 值 + 元数据` 存储。

**典型内容**：个人偏好（语言、语气、格式）、身份信息（部门、工位、联系人）、业务字段（默认项目、常用账号）、关系信息（直属上级、对接客户经理）。

**关键特性**：
- 精准查询（按 key 取）、精准更新（改一条不影响其他）
- 按作用域合并注入：User → App → Team → Org，下层覆盖上层
- 附带类别、置信度、来源等元数据，便于治理
- Agent 可提议，默认进 pending 待用户确认
- 相同 key 再次写入按冲突策略处理（默认高置信度覆盖，旧值归档可回滚）

**和 L1 的区别**：L1 适合"一段连续的描述"，L2 适合"原子级事实"。需要按 key 精准更新的数据不该塞进 markdown。

**类比**：像个人设置面板 + 名片簿。

#### 8.3.4 L3 —— 语义记忆（向量召回）

**定位**：**非结构化的叙述性知识**，一段自由文本 + 它的向量表示，按语义相似度检索。

**典型内容**：过去对话里的重要论断、决策、承诺；用户对某类情况的偏好说明；围绕某主题的零散观点。

**关键特性**：
- **按需出现**：只在用户问题语义相关时才召回 top-K 条，不每次都注入
- 保留原话的自然表达，不强行结构化
- 相似度高于阈值的重复内容自动合并
- 被引用次数多的条目降低衰减速度

**和 L2 的区别**：L2 是"键值事实"按 key 精准取；L3 是"非结构化叙述"按语义相近度取。需要用整段话表达、找不到合适 key 的内容属于 L3。

**和 RAGFlow 的区别**：RAGFlow 检索公司文档（客观知识，人人相同）；L3 检索用户与 agent 的交互上下文（个性化历史）。两者互补——RAGFlow 答"退款政策是什么"，L3 答"上次你对退款流程有什么意见"。

**类比**：像你和助理的工作日记，记下每次拍板和态度，下次相关话题翻出来参考。

#### 8.3.5 L4 —— 情景记忆（会话摘要）

**定位**：**时间点上的事件**，每次对话结束后自动生成一句话摘要 + 向量，每个会话至多一条。

**典型内容**：每次对话的主要议题 + 结论 + 下一步承诺。

**关键特性**：
- 只有**系统后台**写（便宜模型摘要任务），用户和管理员不能直接写（避免篡改系统总结）
- 用户可以**删除**某条摘要（不想留这个会话的痕迹）
- 新会话首条消息时检索最相关的旧会话摘要，相似度高于阈值才召回
- 跟随会话归档 / 被遗忘权一起清理

**和 L3 的区别**：
- L3 是"**事实 / 观点**"——没有时间维度，是用户关于某事的持久态度
- L4 是"**时间点上的事件**"——这次对话发生了什么、什么时候发生

用户问"上次我们聊的方案怎么推进了"——L3 找不到（没有这条事实），L4 能根据语义找到那次对话的摘要，agent 接着聊。

**类比**：像每次会议的会议纪要——议题 + 结论 + 下一步，而不是完整录音。

#### 8.3.6 L5 —— 反思记忆（Agent 自动学习）

**定位**：**Agent 自动抽取的候选事实**，默认处于待确认状态，由用户决定是否接受。

**运作流程**：

```
对话中，用户说了一些关于自己的信息
            │
            ▼
  对话结束，反思引擎用轻量模型读最近几轮消息
  抽取 0-3 条候选事实（带置信度）
            │
            ▼
  写入 L2 / L3 的 pending 区（不参与注入）
            │
            ▼
  前端给用户看小卡片："AI 想记住这些"
            │
            ├─ 用户确认 ──▶ 晋升到 L2 / L3（置信度提升）
            └─ 用户忽略 ──▶ 归档
```

**为什么独立列一层**：Agent 自动提取记忆是企业最敏感的一环——抽错会污染记忆库、抽多会泄漏隐私。L5 不直接写入生效层，单独圈一个"观察区"，**用户是记忆系统的最终仲裁者**。

**治理约束**：
- **成本控制**：按配额触发（如会话前 3 轮、每会话最多抽 3 条），不是每次都抽
- **高敏过滤**：候选包含 PII 敏感信息时直接丢弃，不走 pending（避免敏感信息停留）
- **冷启动阈值**：对话太短（< 4 条消息）不抽取
- **拒绝样本回流**：用户拒绝过某类抽取后，同类再次抽取时权重降低

**类比**：像助理下班前交"今日观察到的用户偏好"清单——助理观察但不擅自决定，你批准的部分才进档案。

#### 8.3.7 五层的分工与合力

五层不是取代关系，而是**分工**。成熟的企业 agent 几乎总是五层并用：

| 层 | 保证的价值 |
|---|---|
| L1 | **规则一致**：全公司 agent 懂术语、知道流程 |
| L2 | **个性一致**：你的偏好在任何应用里都被尊重 |
| L3 | **上下文一致**：你过去说过的重要话被记得 |
| L4 | **时间一致**：能像老朋友一样接续过去的谈话 |
| L5 | **持续成长**：用得越多，agent 对你越熟悉 |

没有 L1，agent 每次得从零自我介绍；没有 L2，用户每次得强调偏好；没有 L3/L4，跨会话失忆；没有 L5，agent 永远停留在初遇版本。**五层合起来，才让 agent 能像资深同事一样和你共事**。

### 8.4 作用域层级

```
Org（组织） ─ Team（团队） ─ App（应用） ─ User（用户）
    │                │             │           │
    ▼                ▼             ▼           ▼
 全租户共享     某团队共享    某应用内共享   仅本人可见
```

查询按 **User → App → Team → Org** 顺序合并；下层同键覆盖上层（越接近个人，越有最终话语权）。

会话级（conversation scope）的记忆不在本层承担，由会话层的运行态快照解决。

### 8.5 读路径（注入）

```
用户消息进入
    │
    ▼
注入引擎
    │
    ├─ L1：按作用域层级加载固定内容
    ├─ L2：按用户 → 应用 → 团队 → 组织合并键值对
    ├─ L3：以用户问题为 query 做向量相似度检索
    ├─ L4：在历史会话摘要里找相关情景
    │
    └─ 按 token 预算裁剪、按重要性（置信度、引用频次、作用域距离）排序
         │
         ▼
    拼装到系统提示词的记忆段落
```

**降级路径**：向量服务 / embedding 网关故障时自动跳过 L3/L4，仅注入 L1/L2，不阻断主对话。

### 8.6 写路径

企业记忆的写入只允许三条路径，所有路径都有明确的治理：

```
路径一：用户主动写 ─────────────▶ 直接 confirmed 状态落库
                                   置信度 0.9+

路径二：Agent 调记忆工具 ───▶ 策略治理 ──▶ 待确认状态
                                              │
                              用户点击确认 ─▶ confirmed + 置信度提升
                              用户点击忽略 ─▶ 归档

路径三：系统后台维护 ─────────▶ 情景摘要自动落库
                                 去重合并 / 置信度衰减
```

**设计要点**：本层的写入收敛到自研治理通道，不复用框架默认的"让模型直接改虚拟文件"的方式——那条路径会绕过 PII 分级、绕过用户确认、绕过审计，不适合企业。

### 8.7 反思流程

```
每轮对话结束
    │
    ▼
反思抽取
    │ 轻量模型从最近几条消息抽取候选事实
    │ 冷启动阈值与调用配额控制成本
    │
    ▼
写入 L2 / L3 候选区（待确认）
    │
    ▼
前端小卡片："AI 想记住这些" → 用户确认 / 忽略
```

### 8.8 记忆条目的生命周期

```
候选 ──(用户确认)──▶ 已确认 ──(持续被引用)──▶ 高置信度记忆
  │                     │
  │                     ├─(长期不被引用)──▶ 置信度衰减 ──▶ 低置信度归档
  │                     │
  │                     ├─(用户主动删)────▶ 归档
  │                     │
  │                     └─(被遗忘权)──────▶ 内容抹除（保留审计空壳）
  │
  └──(用户忽略 / 冷启动阈值)──▶ 归档
```

"归档"不是物理删除，而是保留 N 天以备回滚；真正物理清除要到保留期结束或触发被遗忘权。

### 8.9 治理与合规

- **PII 分级**：普通 / 高敏；高敏默认禁止 agent 自动写、禁止自动注入、审计脱敏展示
- **被遗忘权**：按用户批量抹除其所有作用域内容，审计保留空壳（能证明"抹过"但不保留原文）
- **数据携带权**：用户可导出本人可见的全部记忆
- **多租户**：统一的租户隔离规则，跨租户读写拒绝
- **证据链**：每条记忆可回溯到"最初由哪条消息提出"，用户对记忆有疑义时可跳转原对话

### 8.10 与其他层的协作

| 其他层 | 协作关系 |
|---|---|
| Session Layer | 会话归档时触发情景摘要生成；反馈差评可作为记忆质量调优的参考 |
| Policy Layer | 所有 agent 调用的记忆工具都经过策略治理；高敏记忆的读取可以要求审批 |
| Skill Layer | 技能与记忆边界清晰：技能是"怎么做"，记忆是"你和团队的事实" |
| RAGFlow | 互补关系；RAGFlow 承接企业文档检索，可作为记忆层之外的工具供 agent 按需调用 |

## 9. Skill Layer —— 技能演化与成长

### 9.1 设计定位

技能（Skill）是 agent 能力的复用单元：描述"做某类任务的方法"。现有 easy-ai 只把技能当成静态的提示文件 —— 写一次、用一次、坏了就没人知道。Skill Layer 要把技能从"静态资产"升级为**能看见使用、能测出质量、能随使用长出新版本的数字资产**。

**三个对比可以讲清这层的价值**：

| 对比 | 没有本层 | 有了本层 |
|---|---|---|
| 技能效果 | 凭感觉，只知道"好像能用" | 有成功率、激活率、反馈率等量化指标 |
| 技能维护 | 失败只能靠用户投诉 | 失败自动聚类、LLM 产出改进提案、人工评审 |
| 技能上线 | 手动改好就发 | 评估回归过门禁，影子模式观察，灰度放量 |
| 技能个性化 | 一套 prompt 给所有人 | 团队 / 用户 / 应用可定制变体与个性化覆盖 |

│  │ ·统计判胜     │  │ ·Comment      │  │ ·RBAC         │  │
│  │ ·晋升 / 回滚  │  │ ·跨租户安全   │  │ ·审计         │  │
│  └───────────────┘  └───────────────┘  └───────────────┘  │
└───────────────────────────────────────────────────────────┘
```

每个子系统各司其职，共同驱动"编辑 → 评估 → 放量 → 观测 → 学习 → 再演化"的飞轮。

### 9.3 能力清单

| 能力 | 目的 |
|---|---|
| **生命周期** | 草稿 → 待审 → 影子 → 灰度 → 正式 → 废弃 |
| **使用遥测** | 每次注入 / 激活可追溯，激活全量、纯注入可采样 |
| **评估闭环** | 每个技能有断言集合（工具序列 / 输出 / LLM 判官），修改必跑回归 |
| **变体实验** | A/B 分流、统计显著性判胜、自动 / 手动晋升 |
| **自动演化** | 失败聚类 → LLM 产出改进提案 → 人工评审 → 新变体 → 评估 → 晋升 |
| **组合依赖** | 技能引用技能、版本约束、循环检测 |
| **个性化** | 用户 / 团队 / 应用可覆盖；多语言；Fork |
| **市场协作** | 可见性分级、Star / Fork / Comment |
| **治理** | 签名防篡改、内容扫描、RBAC、审计 |
| **性能** | 编译缓存、token 预算截断、健康度大盘 |

### 9.4 成长飞轮

```
作者编辑技能
    │
    ▼
评估闭环（门禁）
    │ 断言通过率达标 + 关键断言无失败
    │
    ▼
生命周期晋升（影子 → 灰度 → 正式）
    │
    ▼
Agent 运行时调用
    │ 技能编译：解析变体 / 个性化覆盖 / 依赖，带缓存，注入给运行时
    │
    ▼
遥测采集
    │ 激活 / 纯注入多信号归因，明细 + 运行级快照 + 日聚合
    │
    ▼
失败聚类（后台任务）
    │
    ▼
LLM 产出改进提案（进入待审核）
    │
    ▼
人工评审 → 新变体 → 回到评估闭环
```

### 9.5 技能生命周期

```
草稿 ──提交评审──▶ 待审 ──通过──▶ 影子 ──过门禁──▶ 灰度 ──过门禁──▶ 正式
  ▲                                    │                  │           │
  │                                    │                  │           │
  └──拒绝 ──────────────────────────────┘                  │           │
                    失败门禁 ───────────────────────────────┴───────────┘
                    │
                    ▼
                  回到草稿

  正式 ──新版本发布──▶ 废弃（旧版本） ──保留 N 天──▶ 物理删除
  正式 ──手动归档 ──▶ 废弃
```

**每一步晋升都有量化门禁**：

- 进入**影子**：已有评估断言集合、过了内容安全扫描
- 进入**灰度**：评估通过率达标（含关键断言 100% 通过）、影子期样本量足够、无关键回归
- 进入**正式**：灰度期 A/B 指标达标、成功率不回落、反馈率正向

门禁不通过的技能可以回到草稿重改，也可以被审核人拒绝。任何状态迁移都留审计，支持一键回滚上一个版本。

### 9.6 多变体与实验模型

同一技能可以同时存在多个变体参与 A/B，运行时按一致性哈希把用户稳定路由到某个变体（避免同一用户在对话中途换版本）。

**实验指标**：
- **激活率**：技能被注入后是否真的被 agent 用上
- **成功率**：结合用户反馈和工具调用成功率
- **延迟与成本**：响应延迟、token 消耗
- **反馈率**：正负反馈的比例

**判胜机制**：样本量达标 + 统计显著性达置信度阈值后才判胜；胜出变体可手动晋升或配置为自动晋升，老变体归档。实验不收敛的强制期限到期后统一决策或归档。

### 9.7 个性化与继承

```
技能主版本（组织级）
    │
    ▼
变体选择（A/B 决定用哪条主线）
    │
    ▼
个性化覆盖（从大到小依次继承与合并）
    │
    ├─ 团队级覆盖（某团队的差异化提示）
    ├─ 应用级覆盖（某应用的专用语气）
    └─ 用户级覆盖（特定用户的偏好）
    │
    ▼
最终技能内容注入 agent
```

每层覆盖可以只改部分字段（只改提示词 / 只改工具绑定 / 只改语言版本），未覆盖的字段继承上一层。需要深度定制的团队可以 Fork 出独立技能，与上游解耦，上游更新时前端可提示"上游有新版"并支持选择性同步。

### 9.8 市场与协作

**可见性分级**决定技能的暴露范围：

| 级别 | 可见范围 | 可用范围 |
|---|---|---|
| 私有 | 仅所有者 | 仅所有者 |
| 团队 | 所属团队成员 | 所属团队 |
| 组织 | 同租户所有用户 | 同租户 |
| 公开 | 跨租户市场 | 跨租户（按 fork 协议） |

**协作动作**：Star 收藏、Fork 衍生、Comment 评论、Issue 反馈问题。跨租户 Fork 时默认复制到本租户 owner 下，不直接引用原租户资产，防止上游被恶意修改影响下游。公开发布必须过安全扫描（内容注入 / PII / 外链白名单）。

### 9.9 技能健康度

每个技能在管理后台有独立的健康度视图，指标包括：

- **使用维度**：注入次数、激活次数、被应用绑定数、用户收藏数
- **质量维度**：成功率、反馈正负比、失败聚类 Top 模式、关键断言通过率
- **成本维度**：注入 token 中位数、关键运行延迟
- **演化维度**：待审核提案数、活跃变体数、上次晋升时间

组织级健康度大盘按颜色分层：
- 红色：近 24 小时有关键断言失败的技能
- 黄色：通过率下降、反馈率恶化的技能
- 绿色：指标稳定的技能
- 灰色：长期未被使用的技能（建议归档）

### 9.10 与其他层的协作

| 其他层 | 协作关系 |
|---|---|
| Session Layer | 反馈闭环的差评是技能质量调优的关键输入；运行标识是 A/B 归因的主键 |
| Policy Layer | 绑定高风险工具的技能发布前必须过策略评审；技能变更走类审批流程 |
| Memory Layer | 技能关心"怎么做"，记忆关心"你和团队的事实"，两层在提示词段落里并列不交叠 |
| 观测 | 技能的遥测与评估结果统一沉淀到观测平台，方便跨技能对比与长期趋势分析 |

## 10. 共享底座 —— AgentRunContext · Audit · Observability

### 10.1 AgentRunContext

全链路单一上下文，由入口层统一构造：

| 字段 | 含义 |
|---|---|
| run_id | 一次 agent 执行的全局唯一 id（幂等 / 关联 / 观测的主键） |
| app_id | 应用 id |
| user_id | 调用者 |
| conversation_id | 业务会话 id |
| thread_id | 运行态快照的恢复单元，通常与会话 id 绑定 |
| tenant_id | 租户（按项目统一的隔离规则） |
| team_ids | 用户所属团队列表 |
| role_codes | 用户角色列表 |
| trace_id | 观测追踪 id（流结束后回填） |
| request_type | 'chat' / 'api' / 'eval' / 'resume' |

所有中台服务从 `AgentRunContext` 取字段；团队与角色由入口层预拉，支持缓存与登录时写入身份凭证。

### 10.2 统一审计流（WORM）

- 所有策略决策、审批、破窗、指纹不匹配、DLP 命中、记忆访问、技能晋升、GDPR 删除等都进入审计流
- 审计记录带签名链，离线可验证完整性
- 归档：热表 90 天，冷存储按合规要求
- 分租户分区，便于合规导出

### 10.3 可观测性

| 维度 | 工具 | 关键指标 |
|---|---|---|
| 追踪 | Langfuse | 运行级追踪、每次工具调用、token 分布、反馈打分 |
| 指标 | Prometheus | 审批 SLA、破窗次数、记忆注入延迟、技能健康度 |
| 告警 | PagerDuty / Slack | 破窗超阈、指纹不匹配、数据泄漏、审批积压 |
| 大盘 | Grafana | 全局 agent 健康度、技能 A/B 结果、资源压力 |

## 11. 数据与治理

### 11.1 数据分类

| 类别 | 敏感度 | 保留策略 |
|---|---|---|
| 业务会话消息 | 中 | 跟会话软删 + 归档 N 年 |
| Agent 运行态快照 | 中 | 归档会话 30 天后清理 |
| 审批记录 | 高 | 合规周期内不删（可归档冷存） |
| 记忆（KV / 语义 / 情景） | 因人而异 | 用户控制 + 自动衰减 + 被遗忘权 |
| 策略 | 中 | 版本永久保留 |
| 技能 | 中 | 版本永久 + 废弃 N 天后物理删 |
| 遥测 | 低 | 明细月分区归档，聚合表长保留 |
| 审计 | 高 | 合规要求，不删 |

### 11.2 所有权与权限

- **用户**：拥有自己的会话、反馈、个人记忆、私有技能
- **应用管理员**：管理应用绑定的技能 / 工具 / 策略、查看本应用遥测
- **团队管理员**：管理团队作用域记忆与团队可见技能
- **组织管理员**：组织级技能发布权限、审计导出
- **破窗管理员**：紧急放行权限，独立角色，不与普通管理员合并

### 11.3 多租户

- 所有表和查询从第一天起就带租户标识
- 全项目统一隔离规则，跨租户读写拒绝并告警
- 跨租户 fork 默认复制到本租户，不直接引用原租户资源

## 12. 非功能需求

### 12.1 性能

| 指标 | 目标 |
|---|---|
| 首 token 延迟（非记忆启用）| p95 < 1.5s |
| 首 token 延迟（记忆全启用）| p95 < 2.5s |
| L1/L2 记忆注入 | p95 < 50ms |
| L3/L4 语义召回 | p95 < 300-800ms（含 embedding） |
| 策略评估 | p95 < 20ms |
| Checkpoint 写入 | p95 < 100ms |
| 审批 API | p95 < 500ms |

### 12.2 可用性

- 前端 SSE 断线可续流（续流端点，按 seq 回放）
- Embedding 服务超时 → 降级跳过 L3/L4
- LLM provider 故障 → 路由切换（LiteLLM 承担）
- Checkpoint 丢失 → 从业务消息表重建最小 state
- Langfuse 不可用 → 本地审计不阻断

### 12.3 安全

- 所有工具调用经过 policy 兜底 wrapper
- PII 脱敏 + DLP 输出扫描 + Prompt Injection 检测
- 高敏记忆默认不自动注入、审计脱敏
- Skill 内容发布前扫描（PII / 注入 / 外链）
- 审计 WORM + 签名链

### 12.4 成本

- Embedding / 摘要 / LLM judge 使用廉价模型档
- Skill 遥测分层存储（明细采样、聚合长保留）
- Checkpoint 独立库，避免业务热点
- 编译缓存（skill / memory injection）

### 12.5 可运维

- 所有能力域有 feature flag（settings 级 + app_config 级）
- 分期放量：单 app → 低风险业务 → 全量
- 一键回滚：关 flag 即可，数据保留
- 生命周期 gates 可量化（eval pass_rate / shadow 样本量 / 回归阈值）

## 14. 风险与未决

### 14.1 已识别风险

| 风险 | 缓解 |
|---|---|
| 恢复执行时工具调用标识不稳定导致审批幂等被破 | 立项验证 + 增加基于"工具名 + 参数指纹"的次级幂等键 |
| 框架自动注入的默认子 agent 绕过策略治理 | 在应用层强制用同名子 agent 覆盖默认项 |
| 预编译 / 远程子 agent 无治理挂接点 | 启用策略的应用默认禁用这两类子 agent，必要时由 task 工具层治理 |
| 向量维度升级不兼容 | 新表双写 + 逐步切换，禁止原地改列 |
| 反思抽取误记噪声 | 冷启动阈值 + 待确认机制 + 用户拒绝样本回流 |
| 破窗被常态化使用 | 每周复盘 + 告警阈值 + 独立角色权限 |
| 技能自动演化产出低质提案淹没维护者 | 置信度阈值 + 每周配额 + 聚类最小样本 |
| 多副本后台任务并发冲突 | 分布式锁 |
| 租户隔离规则不统一 | 项目级单一决定 |

### 14.2 未决决策

1. **租户隔离规则**：空值语义 vs 默认常量，影响所有查询与索引，需全项目锁定一个方案
2. **续流协议选型**：SSE 重订阅按序回放 vs WebSocket —— 决定前端长任务体验
3. **自定义上下文压缩策略的替换方式**：框架不直接支持替换默认行为，后续走 fork、封装还是上游贡献待定
4. **真实多租户落地时机**：决定何时拆分数据库 / schema
5. **破窗审批的组织机制**：合规团队是否接管、SLA、复盘频率
