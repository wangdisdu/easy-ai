

## L1 信息边界

| 需求                    | 评估  | 证据 & 说明                                                                                                                                                                                                                                           |
| --------------------- | --- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 上下文 40% 性能下降、60% 必须精简 | ⚠   | 压缩触发阈值是 **85% of `max_input_tokens`**（harness 文档 Summarization 段），保留 10% 作为最近上下文。**比需求的 60% 阈值晚得多**，偏向"接近满才动手"。阈值可通过 model profile 覆盖，但没有 40/60 两档的内置告警。                                                                                          |
| 重要内容放提示词开头和结尾         | ⚠   | Prompt 组装顺序固定为 10 层：1. custom system_prompt → 2. base → 3. todo → 4. memory → 5. skills → 6. filesystem → 7. subagent → 8. user middleware → 9. HITL → 10. local context。最前是 custom system_prompt，最后是 local/HITL，但**没有"任务目标放最前、当前指令放最后"的自动位置管理**。 |
| 静态在前、动态在后（缓存友好）       | ✅   | 上述顺序天然把静态内容（system_prompt、base、tool 说明）放前面，动态内容（messages、local context）放后面，**对 Anthropic prompt caching 友好**。官方博客《LangChain Deep Agents》明确提到"uses prompt caching to reduce latency and cost"。                                                     |

**L1 小结**：结构上基本合理（分层 prompt、缓存友好），但**压缩阈值偏晚、没有重要内容位置管理**。

## L2 工具系统

| 需求                            | 评估    | 证据 & 说明                                                                                                                                                                                                                                                                                                                                                                                                                               |
| ----------------------------- | ----- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 统一工具接口规范（五要素：身份、执行、校验、权限、展示 ） | ⚠     | 沿用 LangChain `@tool` 抽象（name / description / schema / function），**没有"风险等级""展示方式"等扩展字段**。Middleware 可以 hook 工具调用流程（`wrap_tool_call`），但不是声明式的五要素规范。<br>缺少权限（基于风险等级的审核）<br>                                                                                                                                                                                                                                                              |
| 工具按风险分三级（只读/受控修改/高风险）         | ❌     | **只有 `FilesystemPermission`**（operations: read/write/edit/ls/glob/grep/execute，mode: allow/deny，按路径 glob 匹配），**不是通用的工具风险分级**。`interrupt_on` 可给任意工具配 HITL，但需要用户自己判断哪些高风险，没有内置的 low/medium/high 标签体系。                                                                                                                                                                                                                                   |
| 大输出自动落盘                       | ✅     | `tool_token_limit_before_evict` 默认 **20,000 tokens**（Claude Code 阈值 25K，非常接近），超限自动落盘到 `/large_tool_results/<tool_call_id>`，只保留前 10 行预览 + 文件路径引用。**设计与 Claude Code 一致**。                                                                                                                                                                                                                                                               |
| ~~错误反馈给 Agent 看（自我纠错）~~       | ~~⚠~~ | ~~**实现有缺陷**。Issue [#927](https://github.com/langchain-ai/deepagents/issues/927) 报告 filesystem middleware 的 `_validate_path` 直接 `raise ValueError` 而不是返回错误消息，导致 Agent 直接崩溃而非收到错误反馈。Issue [#947](https://github.com/langchain-ai/deepagents/issues/947) 报告即使错误返回了，Agent 也可能不自我修正而是直接甩给用户。LangGraph 默认的 `_default_handle_tool_errors` 是 re-raise，需要用户手动配置 `handle_tool_errors=True` 或在工具内自己 try-except。<br><br>~~错误反馈给 Agent 看（自我纠错）~~~~ |
| 工具权限可见性                       | ❌     | 需要我们工程上解决                                                                                                                                                                                                                                                                                                                                                                                                                             |

```
这是 Agent 系统和传统软件系统**最根本的区别之一**：

- **传统软件**：工具/函数出错 → 抛异常 → 程序崩溃或用户看到错误
- **Agent 系统**：工具出错 → 把错误消息作为"下一轮输入"注入回模型 → **模型看到错误、理解问题、自己改正再试**

关键点：**错误不是程序的终点，而是模型推理的新输入**。
```

**L2 小结**：**风险分级和错误反馈机制是两块主要短板**。文件系统权限做得不错，但通用工具的风险管理、错误自修复需要使用者补齐。

## L3 执行编排

| 需求                    | 评估  | 证据 & 说明                                                                                                                                                                                                                                                                                                                                                                                                                                  |
| --------------------- | --- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| TAOR 循环（思考→行动→观察→重复）  | ✅   | 底层是 LangGraph 的标准 tool calling loop，"trust the LLM" 模式由模型自己决定继续或结束。                                                                                                                                                                                                                                                                                                                                                                      |
| 三道止损闸（最大轮数+最大成本+最长时间） | ❌   | **deepagents 最明显的生产短板之一**。官方论坛帖《How to cap tool and sub-agent calls in DeepAgents?》确认：deepagents 本身**没有内置任何止损闸**，只能依赖 LangGraph 的 `recursion_limit`（默认 25，全局超步数限制）。**没有最大成本上限、没有最长运行时间、没有最大工具调用次数**。官方给出的方案是用户自己在 state 里加 `tool_calls` / `subagent_calls` 计数器 + 条件边。生产环境必须自行实现 Middleware 或 Wrapper。<br><br>-----<br>**三道止损闸 = 三个互相独立的硬性上限（最大轮数 / 最大成本 / 最长时间），任意一个先到达就强制停止 Agent 执行**——防止死循环、烧钱、卡死三类失控场景，是 Agent 进生产必须配的"保险丝"。<br><br> |
| 错误不抛出，可恢复的注入回模型       | ⚠   | 理论上 LangChain `ToolNode` 支持 `handle_tool_errors`，但 deepagents **默认配置下 re-raise**（见 L2 的 Issue #927）。需要显式开启。<br><br>----<br>**工具出错时不让程序崩溃，而是把错误消息作为下一轮输入塞回给模型看，让模型自己理解错误并改正重试**——把错误从"程序终点"变成"模型推理的新输入"。                                                                                                                                                                                                                                  |
| 上下文压缩要主动（留余量）         | ❌   | 如前所述，**85% 触发**，跟"剩 30% 空间就压缩"正好相反，属于"等油箱见底才加油"的配置。                                                                                                                                                                                                                                                                                                                                                                                      |
| 压缩是结构化任务（带清单）         | ✅   | **做得很好**。harness 文档明确说明压缩生成包含 **session intent, artifacts created, next steps** 三要素的结构化摘要，且**原始完整对话会同步写入文件系统**作为 canonical record。严格的结构化压缩，不是开放式总结。                                                                                                                                                                                                                                                                                      |
| 子 Agent 跑独立循环，只返回摘要   | ✅   | `task` 工具官方设计：subagent 是 stateless 的一次性实例，有独立 context，autonomous 执行到完成，**只返回单个 final report** 给主 agent。可为不同 subagent 配不同工具和 `interrupt_on`。                                                                                                                                                                                                                                                                                              |

**L3 小结**：**止损闸是一等生产级短板**。压缩时机偏晚，但压缩质量很高。子 Agent 设计优秀。

## L4 记忆与状态 ★ 差距最大

这是 deepagents 相对需求文档**缺失最严重**的一层。需求的核心是"自学习 + 多层记忆 + 策展者"，deepagents 的记忆模型则是"一个虚拟文件系统，Markdown 文件自管理"。

| 需求                         | 评估  | 证据 & 说明                                                                                                                                                                                                                                                                                                                                  |
| -------------------------- | --- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 自学习闭环（执行者/评估者/反思者三角色）      | ❌   | **完全不存在**。deepagents 的 "Memory-First Protocol"（CLI 博客）是让 agent **自己读写 /memories/**：研究时先查、回答前搜索、学习时保存。**没有独立的 Evaluator Agent，没有 Reflexion 式反思机制**。Reflexion 需要用户基于 LangGraph 自己搭建（DeepLearning.AI 有独立课程和社区教程，但**不是 deepagents 内置**）。                                                                                                     |
| 第一步：Reflexion 式反思          | ❌   | 不存在。![[Pasted image 20260419222322.png]]                                                                                                                                                                                                                                                                                                 |
| 第二步：启发式池（带适用条件）            | ❌   | 不存在。<br><br>**把零散的 Reflexion 反思文本，整理成结构化的"经验条目"**——每条都明确标注**"什么情况下适用"+"推荐动作"+"历史成功率"**，形成一个可检索、可评估、可淘汰的经验数据库。<br><br>这是需求文档 L4 自学习的"第二步"，在积累了 100 条左右反思后升级的版本。                                                                                                                                                                           |
| 第三步：技能演化（agent 改写 skill）   | ❌   | Skills 是**只读的人工编写 Markdown 目录**，agent 可以按 progressive disclosure 读取，但**没有自动改写 skill 的能力**。                                                                                                                                                                                                                                               |
| 策展者（选择性添加/删除/合并/冲突检测/生命周期） | ❌   | **完全不存在**。`/memories/` 下的文件完全由 agent 自己判断何时写、何时删，没有独立的 Curator Agent、没有质量评分、没有自动清理、没有冲突检测。**这正是需求第 5 条铁律"自学习必须有策展者，否则 -10%"警告的失控场景**。<br><br>---<br>**管理经验池的"质量守门员"**——它不产生新经验，只决定**哪些经验值得留、哪些该删、哪些该合并、哪些互相冲突、哪些到生命周期尽头**。<br><br>可以理解为经验池的"图书管理员 + 质量审核员 + 垃圾清理工"三合一。**没有策展者，自学习必然越学越坏**——这是需求文档第 5 条铁律的核心结论，也是业界反复实测验证的 **-10% 性能陷阱**。 |
|                            |     |                                                                                                                                                                                                                                                                                                                                          |
|                            |     |                                                                                                                                                                                                                                                                                                                                          |

**L4 小结**：**deepagents 的"记忆"本质上只是一个可持久化的 Markdown 文件柜**，不是认知心理学意义上的分层记忆系统，更没有自学习闭环。**AIOps 场景若需要"越用越好"的能力，L4 需要完全自建**。

## L5 评估与观测

deepagents 本身几乎没有观测能力，**全部依赖 LangSmith**（LangChain 同源商业产品）。

| 需求               | 评估    | 证据 & 说明                                                                                                                                                       |
| ---------------- | ----- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Agent 级追踪（完整轨迹）  | ✅     | LangSmith 提供 full trace，端到端查看多次 LLM 调用 + 多次工具调用的层级关系。官方博客《Evaluating Deep Agents: Our Learnings》详细介绍。                                                         |
| 三类指标（运营/能力/质量）   | ⚠     | LangSmith 原生支持运营指标（latency P50/P99、tokens、cost breakdown）。能力指标（任务完成率、工具失败率）和质量指标（LLM-as-judge、user feedback scores）**需要用户自己定义 evaluators 和 datasets**，不是开箱即用。 |
| 启发式命中率           | ❌     | 由于 L4 没有启发式池，**也就没有这个指标可采集**。                                                                                                                                 |
| ~~单会话成本上限 + 告警~~ | ~~❌~~ | ~~LangSmith 是**事后查看**成本，不是**事中拦截**。deepagents 本身没有 per-session token budget enforcement。这正是《$47,000 Agent Loop》案例警告的场景——**告警不等于预算执行**。~~                      |
| 轨迹数据 SQL 查询      | ✅     | LangSmith 支持，符合"Agent 轨迹是数据"的趋势。<br><br>本身是有的，tracing                                                                                                         |
| skill执行评估        | ❌     | 根据用户的需求，有没有被执行、有没有准确的按skill步骤执行、有没有按预期返回结果                                                                                                                    |

**L5 小结**：追踪和 SQL 查询靠 LangSmith 可用，但**成本事中拦截、启发式指标、三类指标的能力/质量两端都需要自建**。

## L6 约束、检验与恢复

| 需求                         | 评估  | 证据 & 说明                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                 |
| -------------------------- | --- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 架构规则用代码检查器机械强制             | ⚠   | deepagents 的 Middleware 机制（`before_agent` / `wrap_model_call` / `wrap_tool_call`）理论上可实现检查器，但**没有通用的"架构规则检查器框架"**，需要用户自己写 Middleware。`FilesystemPermission` 是最接近的机制（路径级 allow/deny，first-match-wins），但只限于文件系统。<br><br>-----<br>把 Agent 必须遵守的架构规则写成代码检查器（像 ESLint），Agent 违反规则时检查器直接拒绝并返回带修复指令的错误消息，而不是靠 prompt 里写"你不能这样做"来约束。<br><br>**核心差异**：prompt 里写"不要 X" → 模型可能忽略；检查器拦截 → **它绕不过去**。<br><br>**AIOps 例子**：定义规则"L1 诊断 agent 不允许调用生产环境写操作工具"，检查器在 `wrap_tool_call` middleware 里拦截违规调用，返回 `"Error: L1 agent 不能调用 restart_service（写操作）。请改为调用 diagnose_service_status（只读）或派发给 L2 agent。"`——**规则是机械强制的，错误消息直接告诉 agent 怎么改**。                                                                                                                                                                                |
| 全链路格式校验                    | ⚠   | 工具入参用 LangChain tool schema（Pydantic）校验，**但记忆写入、Agent 输出没有 schema 校验**。`/memories/` 就是自由 Markdown。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| **生成器和评估器必须分离**            | ❌   | **L6 最大的缺失**。deepagents **没有独立的 Evaluator Agent 内置**。HITL 可以让人评判，但没有"用 Opus 评 Sonnet 输出"这种 LLM-as-judge 的分离架构内置。需要用户把 subagent 配成 evaluator 角色自己搭。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| 大模型评判器                     | ❌   | 同上。LangSmith 的评估器虽然能做大模型评判，但是它是**事后离线评估**或**抽样评估**，不是**在线实时拦截**。<br><br><br>我觉得事后评估也是没有问题的，但是问题是我们的企业内网版本不能使用langsmith，还是要自己解决。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                         |
| 多策略恢复（至少 4 种）              | ⚠   | deepagents 提供**两种机制**：offloading 和 summarization。<br><br>![[Pasted image 20260420084535.png]]<br>缺少：<br>1. 工具返回的大结果，要先摘要，再落盘。<br>2. 摘要压缩，早期对话总结成一段摘要，替换原始消息，但是原始消息依旧要落盘，要能够回查。<br><br>**deepagents 的现状**<br><br>deepagents 只有**第 1 档**的两种做法（落盘卸载 + 摘要压缩），**没有第 2、3、4 档**。换句话说：<br><br>- 上下文满了 → 它会压缩（有）<br>- 压缩了还是不行 → **直接崩，没有降级任务难度的机制**<br>- 工具调用一直失败 → **没有"换个策略再试一次"的逻辑**<br>- 走到死胡同 → **没有优雅报错的机制**（上面讲过连止损闸都没有）                                                                                                                                                                                                                                                                                                                                                                                  |
| 关键操作前打检查点，失败可回滚（针对沙盒操作可回滚） | ⚠   | LangGraph `checkpointer` 保存 **graph state**（messages、file contents in state），**但不保存 sandbox 的真实文件系统快照**。sandbox backend（Modal/Daytona）下的 shell 副作用（`rm`、`DROP TABLE`）**不在 checkpoint 里**，无法回滚。**这个陷阱跟需求文档明确警示的完全一致**。<br><br>langgraph的checkpoint是应用层状态快照，不是文件系统快照。对状态之外的真实问题，无法解决。示例：<br><br>```<br>[第 1 步] Agent: 我需要清理过期日志 调用 execute("ls /var/log/") checkpoint_1 存下来 ✅ <br>[第 2 步] Agent: 看起来 prod_app/ 下有很多日志 调用 execute("rm -rf /var/log/prod_app/*.log") ← 真实删除了 200GB 生产日志 checkpoint_2 存下来 ✅（但只存了 messages，没存磁盘状态）<br>[第 3 步] Agent: 咦，应该先 rotate 的，删错了 想回滚到第 1 步 [试图恢复] agent.invoke(input, config={"configurable": {"thread_id": "xxx", "checkpoint_id": "checkpoint_1"}}) 结果: ✅ messages 回到第 1 步状态（对话历史恢复） ✅ TODO 列表回到第 1 步 ❌ /var/log/prod_app/*.log 还是被删了 ← 灾难 ❌ 200GB 日志找不回来<br>```<br> |
| 高风险操作强制人工审批                | ✅   | `interrupt_on={"tool_name": True}` 直接支持，且三选一（approve/edit/reject）可配置，subagent 可继承或覆盖父规则。**deepagents L6 最成熟的能力**。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |
| 核心规则硬编码，反思不能覆盖             | ❌   | 由于 L4 没有自学习机制，这一条是 N/A。但反过来说，如果未来加上自学习，**deepagents 没有内置的"不可变规则"保护区**。<br><br>![[Pasted image 20260420090514.png]]                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                      |
| 重要技能修改需人工通过                | ❌   | Skills 目前是人工编写的静态目录，agent 无法修改。如果未来允许 agent 写 skill，**没有人工审批闸口**。                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                                       |

**L6 小结**：`interrupt_on` 是唯一做得好的。**生成器/评估器分离、多策略恢复、检查点回滚（特别是 sandbox 副作用）、自学习约束全部缺失**。
