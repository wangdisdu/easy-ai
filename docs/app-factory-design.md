## 1. 功能概述

应用工厂是平台面向业务侧输出 AI 能力的核心模块，负责承载应用的创建、配置、测试、发布、下线、调用接入与运行态观察。其目标不是单纯保存一份“应用配置”，而是将平台底层的大模型、知识库、工具、技能、编排能力以应用化方式封装给用户和外部系统使用。

应用工厂覆盖五类应用：

| 应用类型 | 说明 | 底层能力 |
|------|------|------|
| `rag` | 基于知识库的检索增强问答应用 | RAGFlow |
| `llm` | 通用大模型应用，支持提示词编排、输入输出变量定义 | LiteLLM |
| `nl2sql` | 自然语言转 SQL 查询应用 | NL2SQL Runtime |
| `agent` | 可调用工具与技能的智能体应用 | DeepAgents / Agent Runtime |
| `agent_flow` | 多节点、多能力编排的流程型应用 | Flowise / Flow Engine |

应用状态统一分为三类：

| 状态 | 含义 |
|------|------|
| `draft` | 草稿，仅平台内可编辑和测试，不能对外提供正式访问 |
| `published` | 已发布，可供授权用户与外部系统正式调用 |
| `offline` | 已下线，不再对外提供服务，但配置和历史记录保留 |

应用工厂的设计原则如下：

- 面向模板化创建：用户基于应用类型快速开始，而不是从零配置底层能力。
- 面向配置分层：基础信息、模型参数、业务配置、平台配置分区组织，降低复杂度。
- 面向运行闭环：创建后可测试、发布、查看会话、查看配置、查看 API 文档。
- 面向后续扩展：详情页预留会话记录、可观测数据、API 文档等运行态入口。

## 2. 页面范围

应用工厂涉及 3 个核心页面：

| 页面 | 原型文件 | 作用 |
|------|------|------|
| 应用列表页 | `AppListView.vue` | 展示全部应用，支持检索、筛选、进入详情与创建 |
| 应用创建/编辑页 | `AppFormView.vue` | 按应用类型完成创建或编辑配置 |
| 应用详情页 | `AppDetailView.vue` | 查看已创建应用的概览、配置、会话、API 文档与发布操作 |

此外，Agent Flow 在原型中采用“新窗口进入可视化编辑器”的方式，属于创建页的扩展形态，不单独拆页面。

## 3. 应用列表页设计

### 3.1 页面目标

应用列表页承担“应用总览入口”的角色，解决以下问题：

- 快速看到系统中已有的全部应用
- 按类型和状态筛选应用
- 按名称和描述搜索应用
- 快速进入创建页或某个应用详情页
- 在列表维度完成删除操作

### 3.2 页面结构

页面结构由上到下分为 4 个区域：

1. 页面标题区
2. 搜索与筛选区
3. 应用卡片网格区
4. 空状态 / 删除确认弹窗

### 3.3 标题区

- 标题：`应用工厂`
- 副标题：`基于模板快速创建 LLM / RAG / NL2SQL / Agent / Agent Flow 应用`
- 主操作按钮：`创建应用`
- 点击“创建应用”进入创建页

### 3.4 搜索与筛选区

列表页支持三类筛选能力：

| 筛选项 | 说明 |
|------|------|
| 关键词搜索 | 按应用名称或描述模糊匹配 |
| 类型筛选 | 全部 / RAG / NL2SQL / LLM / Agent / Agent Flow |
| 状态筛选 | 全部状态 / 已发布 / 草稿 / 已下线 |

其中类型筛选按钮需要显示各类型应用数量，帮助用户感知分布。

### 3.5 应用卡片

每个应用卡片展示以下信息：

| 字段 | 说明 |
|------|------|
| 应用类型标签 | 不同类型不同颜色 |
| 状态标识 | 圆点 + 状态文字 |
| 应用名称 | 卡片主标题 |
| 应用描述 | 最多两行截断 |
| 创建时间 | 卡片底部左侧 |
| 累计调用次数 | 卡片底部右侧 |

建议的颜色语义与原型保持一致：

| 类型 | 视觉色 |
|------|------|
| `RAG` | 蓝色 |
| `NL2SQL` | 青色 |
| `LLM` | 绿色 |
| `Agent` | 紫色 |
| `Agent Flow` | 琥珀色 |

### 3.6 交互行为

- 点击卡片主体：进入应用详情页
- 鼠标悬停卡片：显示删除按钮
- 点击删除按钮：弹出二次确认
- 删除后：从列表移除，卡片消失
- 无匹配数据时：展示空状态和提示文案

### 3.7 列表页接口建议

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET` | `/api/v1/app/page` | 分页获取应用列表，支持 `keyword`、`app_type`、`app_status` |
| `DELETE` | `/api/v1/app/{id}` | 删除应用 |

分页接口建议返回简版字段，不必携带所有配置详情。

## 4. 创建/编辑页设计

### 4.1 页面目标

创建页需要解决以下问题：

- 先选应用类型，再进入对应类型的配置表单
- 在同一个页面中兼容“创建”和“编辑”两种模式
- 支持按应用类型展示差异化配置
- 提供随时测试和保存能力
- 为 Agent Flow 提供可视化编排入口

### 4.2 整体流程

创建页采用 2 步模式：

```text
Step 1 选择应用类型
Step 2 配置应用
```

编辑模式下：

- 页面通过 `edit` 与 `type` 参数直接进入 Step 2
- 页面标题改为“编辑应用”
- 返回按钮回到应用详情页

### 4.3 Step 1：选择应用类型

选择区以卡片方式展示 5 种应用类型，每张卡片包含：

- 图标
- 名称
- 说明
- 特性标签
- 类型主色

| 类型 | 说明摘要 | 特性标签示例 |
|------|------|------|
| Agent | 具备自主决策能力，可调用工具完成多步任务 | 多步推理、工具调用、技能编排、自主决策 |
| Agent Flow | 可视化工作流编排 | 可视化编排、节点拖拽、条件分支、并行执行 |
| LLM | 通用大模型应用 | 多模型切换、Fallback、统一接口、流式输出 |
| RAG | 检索增强知识问答 | 语义检索、Prompt 拼接、引用溯源、多知识库 |
| NL2SQL | 自然语言数据查询 | Schema 检索、SQL 生成、自动执行、图表输出 |

交互规则：

- 点击类型卡片进入 Step 2
- 点击 `Agent Flow` 时，可直接进入 Flow 编辑器模式

### 4.4 Step 2：配置应用

Step 2 是差异化表单区域，但需要共享一套统一结构。

建议统一分为以下层次：

1. 基础信息
2. 模型与推理参数
3. 类型特有配置
4. 平台配置
5. 测试入口与保存操作

### 4.5 共用配置

#### 4.5.1 基础信息

所有应用类型都应包含：

| 字段 | 必填 | 说明 |
|------|------|------|
| `name` | 是 | 应用名称 |
| `description` | 否 | 应用描述 |

页面中直接通过区域标题体现应用类型，不需要用户再次手工选择“应用类型”。

#### 4.5.2 模型与推理参数

适用于 `llm`、`rag`、`nl2sql`、`agent`，不适用于 `agent_flow`。

模型选择需基于平台已接入的大模型供应商与模型配置：

- 先选择供应商
- 再选择模型
- 配置通用推理参数

建议字段如下：

| 字段 | 必填 | 说明 |
|------|------|------|
| `provider_id` | 是 | 供应商 ID |
| `model_id` | 是 | 模型 ID |
| `temperature` | 否 | 温度，默认 0.7 |
| `max_tokens` | 否 | 最大输出 Token |

实现上也可以只在前端展示为一个“模型选择器”，后端最终仍拆为 `provider_id + model_id`。

#### 4.5.3 平台配置

所有应用都应支持平台侧访问控制与运行控制：

| 字段 | 说明 |
|------|------|
| `access_scope` | 访问范围：企业内部 / API 开放 / 嵌入式 |
| `rate_limit` | 限流，单位次/分钟 |
| `enable_log` | 是否保留调用日志 |

### 4.6 LLM 应用配置

LLM 应用用于对话、摘要、提取、分类、报告生成等通用场景。

除共用配置外，需要支持以下区域：

#### 4.6.1 输入变量定义

支持用户动态维护输入变量列表，每项包含：

| 字段 | 说明 |
|------|------|
| `name` | 变量名，英文标识 |
| `label` | 前端展示名 |
| `type` | `text` / `textarea` / `number` / `select` / `file` |
| `required` | 是否必填 |

#### 4.6.2 提示词配置

包含两个区域：

| 字段 | 说明 |
|------|------|
| `system_prompt` | 系统提示词 |
| `user_prompt` | 用户提示词模板 |

提示词中允许通过 `{{变量名}}` 引用输入变量，界面上应提供快捷插入能力。

#### 4.6.3 输出配置

| 字段 | 说明 |
|------|------|
| `output_format` | `text` / `json` / `markdown` |
| `output_schema` | 结构化输出时定义 schema |
| `output_vars` | 原型中体现为输出变量列表，用于辅助定义结果结构 |

文档建议：

- 若 `output_format = text`，不要求 `output_schema`
- 若 `output_format = json`，必须允许配置结构化 schema 或输出变量

### 4.7 RAG 应用配置

RAG 应用重点在知识库选择与检索链路参数配置。

除共用配置外，需要支持以下区域：

#### 4.7.1 知识库绑定

支持选择一个或多个知识库：

| 字段 | 说明 |
|------|------|
| `kb_ids` | 绑定知识库 ID 列表 |

界面上可展示知识库名称、文档数、Chunk 数等摘要信息。

#### 4.7.2 检索参数

| 字段 | 说明 |
|------|------|
| `similarity_threshold` | 相似度阈值 |
| `vector_weight` | 向量检索权重 |
| `top_n` | 召回条数 |

#### 4.7.3 Rerank 配置

| 字段 | 说明 |
|------|------|
| `enable_rerank` | 是否启用 Rerank |
| `rerank_model` | Rerank 模型 |
| `rerank_top_n` | Rerank 后保留数量 |

#### 4.7.4 总结配置

| 字段 | 说明 |
|------|------|
| `enable_summary` | 是否启用总结 |
| `summary_model` | 总结模型 |
| `summary_temperature` | 总结温度 |
| `summary_prompt` | 总结提示词 |

### 4.8 NL2SQL 应用配置

NL2SQL 应用面向结构化数据查询。

除共用配置外，需要支持以下字段：

| 字段 | 说明 |
|------|------|
| `db_connection` | 数据源连接标识 |
| `db_schema` | Schema / DDL 描述 |

如果后续支持更多能力，可继续扩展：

- SQL 安全策略
- 可访问表范围
- 结果集上限
- 图表生成开关

### 4.9 Agent 应用配置

Agent 应用重点在可用工具、技能和行为策略。

除共用配置外，需要支持以下区域：

#### 4.9.1 系统提示词

定义智能体角色、工作原则、行为边界与升级策略。

#### 4.9.2 工具绑定

从平台工具中心tb_tool选择多个工具：

| 字段 | 说明 |
|------|------|
| `tool_ids` | 绑定工具 ID 列表 |

界面需展示工具名称、描述、来源类型。

#### 4.9.3 技能绑定

从平台技能中心tb_skill选择多个技能：

| 字段 | 说明 |
|------|------|
| `skill_ids` | 绑定技能 ID 列表 |

#### 4.9.4 行为参数

| 字段 | 说明 |
|------|------|
| `max_turns` | 最大对话轮次 |
| `agent_timeout` | 执行超时时间（秒） |
| `allow_auto_exec` | 是否允许自动执行高风险动作 |

#### 4.9.5 子智能体

原型中体现为子智能体列表，建议作为扩展配置保留：

| 字段 | 说明 |
|------|------|
| `sub_agents[].name` | 子智能体名称 |
| `sub_agents[].model` | 子智能体使用模型 |
| `sub_agents[].role` | 子智能体职责描述 |

## 5. 创建页测试能力

创建/编辑过程中需支持测试面板或测试抽屉。

### 5.1 测试目标

- 在发布前验证配置是否可运行

Agent 测试界面需要尽量展示中间过程，如：

- 工具调用记录

- 最终响应

## 6. 应用详情页设计

### 6.1 页面目标

应用详情页承载“运行态管理中心”的角色，用户应在这里完成：

- 查看应用当前状态与核心信息
- 查看概览数据与可观测信息
- 查看会话记录
- 查看应用配置
- 获取 API 接入文档
- 进行编辑、测试、发布、删除等操作

### 6.2 页面结构

详情页从上到下包括：

1. 返回按钮
2. 草稿提示横幅
3. 应用头部 Hero 卡片
4. Tab 内容区
5. 测试抽屉 / 发布弹窗 / 删除确认弹窗

### 6.3 应用头部区域

头部卡片展示：

| 信息 | 说明 |
|------|------|
| 应用图标 | 按类型显示 |
| 类型标签 | 颜色与列表页保持一致 |
| 状态标签 | 草稿 / 已发布 / 已下线 |
| 应用名称 | 主标题 |
| 应用描述 | 副标题说明 |
| 元信息 | 创建人、创建时间、引擎、模型、版本、累计调用量 |

操作按钮包括：

- 编辑
- 测试
- 删除
- 发布应用 / 发布新版本

当应用处于草稿状态时，页面顶部应显示提示横幅，强调“仅平台内可测试，尚未对外开放”。

### 6.4 Tab 规则

详情页 Tab 应按状态动态展示：

| Tab | 草稿可见 | 已发布可见 | 说明 |
|------|------|------|------|
| 概览 | 是 | 是 | 运行概况和统计摘要 |
| 会话记录 | 否 | 是 | 历史调用和交互记录 |
| 应用配置 | 是 | 是 | 查看和编辑应用配置 |
| API 及文档 | 否 | 是 | 外部系统接入说明 |

### 6.5 概览 Tab

概览页需要按应用类型展示不同内容。

#### 6.5.1 通用区域

草稿状态下重点展示配置摘要：

- 应用类型
- 运行引擎
- 模型
- Temperature
- Max Tokens
- 关键配置摘要

已发布状态下重点展示运行统计：

| 指标卡片 | 说明 |
|------|------|
| 总调用次数 | 历史累计调用量 |
| 今日调用 | 今日调用次数 |
| 平均延迟 | 平均响应耗时 |
| 成功率 / 满意率 | 根据类型或业务定义展示 |

并展示调用趋势图。

#### 6.5.2 RAG 概览

RAG 应用需要展示：

- 已绑定知识库列表
- RAG Pipeline 耗时分解
- 检索质量趋势
- Chunk 命中明细 Top N
- 空检索高频问题

#### 6.5.3 LLM 概览

LLM 应用需要展示：

- Token 用量趋势
- 延迟分布（P50/P95/P99）
- 错误分类分布

#### 6.5.4 Agent / Agent Flow 概览

Agent 类应用需要展示：

- 工具列表
- 技能列表
- 子智能体列表
- 执行耗时分解
- 工具调用趋势
- 任务状态分布
- Token 趋势

Agent Flow 还应增加流程步骤视图。

### 6.6 会话记录 Tab

会话记录是已发布应用的重要运行态视图。

支持按来源筛选：

| 值 | 说明 |
|------|------|
| `all` | 全部 |
| `platform` | 平台内对话 |
| `api` | API 调用 |
| `test` | 平台测试 |

非 Agent 应用建议字段：

| 列 | 说明 |
|------|------|
| 来源 | 调用来源 |
| 调用方 | 用户或系统 |
| 查询内容 | 用户输入 |
| 响应摘要 | 输出结果摘要 |
| 时间 | 调用时间 |
| Tokens | Token 使用量 |
| 延迟 | 耗时 |
| 反馈 | 正向 / 负向 / 无 |

Agent 应用建议字段：

| 列 | 说明 |
|------|------|
| 来源 | 调用来源 |
| 调用方 | 用户或系统 |
| 对话主题 | 任务摘要 |
| 轮次 | 对话轮数 |
| 步骤 | 执行步骤数 |
| 工具调用 | 调用次数与工具名称 |
| 时间 | 调用时间 |
| Tokens | Token 使用量 |
| 耗时 | 总耗时 |
| 状态 | 成功 / 部分完成 / 超时 |
| 反馈 | 用户反馈 |

支持分页，默认每页 20 条。

### 6.7 应用配置 Tab

应用配置页需支持两种模式：

- 查看模式
- 编辑模式

非 `agent_flow` 类型：

- 默认查看
- 点击“编辑”进入编辑态
- 表单内容与创建页 Step 2 保持一致
- 点击“保存”提交变更
- 点击“取消”回到查看态

`agent_flow` 类型：

- 展示流程摘要
- 提供“进入编排器”按钮

### 6.8 API 及文档 Tab

该 Tab 面向外部系统接入方。

页面分为两部分：

#### 6.8.1 调用示例

- 接入地址
- 请求方法
- Header 示例
- cURL 示例
- 响应示例

#### 6.8.2 字段说明

- 认证方式
- 请求参数说明
- 响应字段说明
- 会话标识说明
- API Key 获取路径说明

建议路由规则：

| 类型 | 路径 |
|------|------|

## 7. 数据模型设计

### 7.1 应用主表

建议主表 `tb_app` 只维护应用通用元数据、模型绑定、通用推理参数与平台控制字段，不再把核心关系字段塞入 `config`：

| 字段                | 类型 | 说明                                                     |
|-------------------|------|--------------------------------------------------------|
| `id`              | bigint | 应用 ID                                                  |
| `name`            | varchar | 应用名称                                                   |
| `description`     | text | 应用描述                                                   |
| `app_type`        | varchar | 应用类型 `rag` / `llm` / `nl2sql` / `agent` / `agent_flow` |
| `app_status`      | varchar | 应用状态 `draft` / `published` / `offline`                 |
| `app_config`      | text nullable | 根据不用的应用类型有不同的schema                                    |
| `provider_id`     | bigint nullable | 模型供应商                                                  |
| `model_id`        | bigint nullable | 绑定模型                                                   |
| `model`           | varchar nullable | 模型名称快照，便于列表页直接展示                                       |
| `model_setting`   | text nullable | 模型设置                                                   |
| `access_scope`    | varchar | 访问范围                                                   |
| `rate_limit`      | int | 限流值                                                    |
| `enable_log`      | tinyint | 是否保留调用日志                                               |
| `version_id`      | varchar nullable | 当前发布版本id                                               |
| `current_version` | varchar nullable | 当前发布版本，便于列表页直接展示                         |
| `create_time`     | bigint | 创建时间                                                   |
| `update_time`     | bigint | 更新时间                                                   |
| `create_user`     | bigint nullable | 创建人                                                    |
| `update_user`     | bigint nullable | 更新人                                                    |

设计原则：

- `app_config` 根据不用的应用类型有不同的schema，例如 RAG 检索参数、Agent 工具绑定、Flow 编排元数据。
- `provider_id`、`model_id`、`model_setting` 属于主表字段，因为它们是筛选、审计、回显和版本快照都高频使用的核心配置。
- `model` 为冗余展示字段，来源于 `tb_llm_model.model`，用于减少列表查询时的联表成本。
- `model_setting` 为模型参数设置，有temperature、max_tokens，未来可能有更多细节控制参数
- `agent_flow` 类型通常不依赖单一模型，因此 `provider_id`、`model_id`、`model`、`model_setting` 可为空。

## 9. 当前实现边界与阶段建议

结合当前代码现状，应用工厂可以分为两个层次理解：

### 9.1 当前已实现

- 应用列表页基础能力
- 创建 / 编辑页基础表单

- API 文档页的动态请求/响应示例生成

### 9.3 推荐落地顺序

## 10. 结论

应用工厂不是单一页面，而是平台 AI 能力产品化输出的核心枢纽。其设计重点在于：

- 通过统一的创建流程屏蔽底层复杂度
- 通过类型化配置表达不同 AI 应用的特征
- 通过详情页构建从配置到运行的闭环
- 为后续知识库、工具、技能、Flow 编排、可观测能力留出清晰扩展位
